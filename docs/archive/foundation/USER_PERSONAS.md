## Purpose

Define who FlowOS is built for, who to recruit for private alpha, and who to **avoid** early (anti-personas). Grounded in [PRODUCT_VISION.md](./PRODUCT_VISION.md) and the 90-day goal of **product validation with real users**.

---

## Primary persona: Alex — Knowledge Worker

### Profile

| Attribute | Detail |
|-----------|--------|
| Age | 28–38 |
| Role | Software developer, researcher, writer, or consultant |
| Context | Remote or hybrid; manages own schedule |
| Current tools | Todoist or Things + habit tracker (Habitica/Streaks) + focus timer (Forest/Toggl) + occasional journaling (Day One/Notion) |
| Tech comfort | High — tolerates beta software if value is clear |

### Jobs to be done

1. **Morning:** See what's on today and start the highest-priority work quickly  
2. **Midday:** Track focus time on real tasks without switching apps  
3. **Evening:** Brief reflection on what worked and what didn't  
4. **Weekly:** Sense of progress across tasks, habits, and focus — without exporting to spreadsheets  

### Pain points (today, without FlowOS)

- Switches between 3–4 apps daily; data fragmented  
- Task app doesn't know about focus sessions or habits  
- Reflection happens in a separate journal — disconnected from actual work data  
- "What should I do right now?" requires mental assembly from multiple sources  

### Why Alex fits FlowOS alpha

- Already has productivity habits — will evaluate the **daily loop**, not basic task creation  
- Will notice routing friction, module switching, and missing command palette  
- Feedback will focus on retention ("would I open this tomorrow?") not feature requests  

### Alpha recruitment criteria

- Uses 2+ productivity apps daily for at least 3 months  
- Willing to use FlowOS as **primary daily driver** for 2 weeks  
- Available for 30-minute onboarding call + 15-minute check-in at day 7  
- Honest feedback style — will say when something feels broken

## Secondary persona: Sam — Graduate Student

### Profile

| Attribute | Detail |
|-----------|--------|
| Age | 22–28 |
| Role | Master's or PhD student; coursework + research + personal development |
| Context | Structured deadlines; irregular deep work blocks |
| Current tools | Notion or Google Calendar + simple todo + habit app |
| Tech comfort | Medium-high |

### Jobs to be done

1. Balance coursework tasks with research habits (reading, writing blocks)  
2. Track focus sessions during thesis/lab work  
3. Weekly reflection for advisor meetings or personal review  
4. Capture ideas and meeting notes without losing daily task context  

### Pain points

- Calendar shows meetings but not habits or focus intent  
- Habit tracking feels gamified but disconnected from academic goals  
- Notes live in Notion; tasks live elsewhere  

### Why Sam fits FlowOS alpha

- Aligns with original thesis audience ([../archive/project/01-introduction.md](../archive/project/01-introduction.md))  
- Needs integrated loop more than feature breadth  
- Good secondary alpha cohort after Alex validates core loop  

### Alpha recruitment criteria

- Manages 5+ recurring responsibilities (courses, research, personal)  
- Already reflects or journals at least weekly  
- Desktop/laptop primary device (web-only constraint)

## Anti-persona: Casey — Casual List-Maker

### Profile

| Attribute | Detail |
|-----------|--------|
| Role | Anyone who needs a simple shopping list or occasional reminders |
| Current tools | Apple Reminders, Google Keep |
| Needs | Add item, check off, forget |

### Why Casey is NOT the target (yet)

- FlowOS complexity (8 modules, planning model, reflection) creates friction without payoff  
- Will not use habits, focus sessions, or reflection — evaluates only task list UX  
- Feedback will request simplification that conflicts with SRL vision  

**Rule:** Do not recruit Casey for alpha. Do not optimize for Casey until daily loop is proven and onboarding exists.

---

## Anti-persona: Dev — Developer Evaluator

### Profile

| Attribute | Detail |
|-----------|--------|
| Role | Engineer reviewing FlowOS as a portfolio piece or tech demo |
| Behavior | Tolerates broken UX; focuses on stack, code quality, feature checklist |

### Why Dev is NOT ideal for alpha (as primary cohort)

- Tolerates missing onboarding and wrong routing — masks real user friction  
- Feedback skews toward feature requests (AI, API, keyboard shortcuts) over daily retention  
- May never attempt a full plan → execute → reflect day  

**Rule:** Developer feedback is valuable for engineering; do not use it as primary product validation signal.

## Alpha cohort composition (recommended)

| Cohort | Size | Persona |
|--------|------|---------|
| Wave 1 | 5 | Alex — knowledge workers |
| Wave 2 | 5–10 | Mix of Alex + Sam |
| Maximum private alpha | 15 | Before expanding to beta |

Recruit personally (direct message, not public launch). Onboard via video call. Watch first 10 minutes of use.

---

## Feedback questions by persona

Ask every alpha user weekly:

1. Did you open FlowOS today without reminding yourself?  
2. How long from open to first meaningful action?  
3. How many times did you think "where am I supposed to be?"  
4. Did you complete reflection at least 3 days this week?  
5. What did you use **outside** FlowOS because FlowOS failed?
